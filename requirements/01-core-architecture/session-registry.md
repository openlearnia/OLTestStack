# Session Registry

> **Module:** `01-core-architecture`  
> **Feature:** In-memory session registry  
> **Depends on:** [types.md](./types.md)  
> **Depended on by:** [02-browser-session-management](../02-browser-session-management/REQUIREMENTS.md), [03-page-session-management](../03-page-session-management/REQUIREMENTS.md), all interaction modules (04â€“11)

## Overview / Purpose

Define the in-memory registry that tracks active `BrowserSession` and `PageSession` instances. The registry is the server-side source of truth for session state while MCP commands remain stateless in their request payloads (clients pass IDs explicitly).

---

## Functional Requirements

### Session Model

| ID | Requirement |
|----|-------------|
| FR-01-006 | The server SHALL maintain an in-memory registry of active `BrowserSession` and `PageSession` objects keyed by UUID. |
| FR-01-007 | Session IDs (`browserId`, `pageId`, `elementId`) SHALL be opaque UUIDs (v4) generated by the server. |
| FR-01-008 | Closing a browser session SHALL cascade-close all associated page sessions and invalidate all element references. |
| FR-01-009 | The system SHALL support multiple `PageSession` instances per `BrowserSession`. |

### Registry Operations

| ID | Requirement |
|----|-------------|
| FR-01-021 | The registry SHALL support create, lookup, delete, and cascade-delete operations for browser and page sessions. |
| FR-01-022 | Lookup of a non-existent session SHALL return a result that maps to `SESSION_NOT_FOUND` at the MCP layer. |
| FR-01-023 | Closing a page session SHALL remove it from the parent browser's `pageIds` list and invalidate all element IDs scoped to that page. |
| FR-01-024 | Element references SHALL be stored in a secondary map keyed by `pageId`; element invalidation SHALL clear entries for the affected page. |

---

## Non-Functional Requirements

| ID | Requirement |
|----|-------------|
| NFR-01-005 | Session registry SHALL be thread-safe / async-safe for concurrent MCP tool invocations on the same session. |
| NFR-01-006 | Memory usage SHALL not grow unbounded; closed sessions and their recordings SHALL be eligible for garbage collection. |

### Cascade-Delete Behavior

| Trigger | Effect |
|---------|--------|
| `browser.close` | Remove browser entry; remove all child pages; invalidate all elements for those pages |
| `page.close` | Remove page entry; remove from parent `pageIds`; invalidate elements for that page |
| `page.navigate` / `page.reload` | Invalidate elements for that page (page entry retained) |
| External browser crash | Mark browser as crashed; subsequent lookups return `BROWSER_CRASHED` |

---

## MCP Command Spec

This feature does not expose MCP commands directly. Registry operations are invoked internally by:

| Command | Registry Operation |
|---------|-------------------|
| `browser.launch` | Create `BrowserSession` |
| `browser.close` | Cascade-delete browser and pages |
| `page.create` | Create `PageSession`, link to browser |
| `page.close` | Delete `PageSession` |
| `page.navigate` / `page.reload` | Invalidate elements for page |
| All commands using IDs | Lookup session by ID |

## Acceptance Criteria

| ID | Criterion |
|----|-----------|
| AC-01-003 | Session registry supports create, lookup, delete, and cascade-delete operations. |
| AC-01-011 | Closing a browser with 3 open pages removes all 4 registry entries. |
| AC-01-012 | Concurrent lookups and closes on the same session do not corrupt registry state. |
| AC-01-013 | Element map entries for a page are cleared after navigation, reload, or page close. |
| AC-01-014 | Closed sessions are not retained in memory after all references are released. |

## Out of Scope (V2)

- Persistent session storage / session resume
- Distributed session registry (multi-process)
- Session TTL / automatic expiry
- Session listing or introspection MCP commands
- Cross-process session sharing
